# Cloud Deploy Conventions

## Environments

| Environment | AWS Suffix | Databricks Catalog | Infra Management | App Deploy |
|-------------|------------|-------------------|-------------------|------------|
| **DEV** | `-dev` | `dev` | Terraform (`services/dev/terraform/1_aws_core`) via `deploy_cloud_infra_dev.yml` | Docker Compose (local streaming), DABs CLI (`databricks bundle deploy --target dev`) |
| **HML** | `-hml` | `hml` | Terraform (`services/hml/1_aws_core`) via `deploy_cloud_infra_prd.yml` + ephemeral resources in CI/CD | DABs CLI via `deploy_databricks.yml`, ECS RC image via `deploy_streaming_apps.yml` |
| **PROD** | `-prd` | `dd_chain_explorer` | Terraform (`services/prd/`) via `deploy_cloud_infra_prd.yml` | DABs CLI via `deploy_databricks.yml`, ECS via `deploy_streaming_apps.yml` |

## Infrastructure (Terraform)

- **All AWS infrastructure** is managed by Terraform with remote S3 backend (`dm-chain-explorer-terraform-state`).
- **DEV infra** lives in `services/dev/terraform/1_aws_core/` — a single flat module with S3, DynamoDB, Kinesis, SQS, CloudWatch Logs, Lambda.
- **HML infra** lives in `services/hml/1_aws_core/` — persistent resources (S3 bucket, IAM roles, CloudWatch Logs + Firehose). Ephemeral resources (ECS cluster, Kinesis streams, SQS queues, DynamoDB table) are created/destroyed per CI/CD run.
- **PROD infra** lives in `services/prd/` — modular numbered layout: `0_remote_state`, `1_vpc`, `2_iam`, `3_kinesis_sqs`, `4_s3`, `6_ecs`, `7_databricks`, `9_dynamodb`, `10_lambda`. No `terraform/` sub-folder — each module is at `services/prd/<module>/`.
- Shared Terraform modules live in `services/modules/` (kinesis, sqs, cloudwatch_logs).
- **Never** create AWS resources manually or via CLI for any environment. All infra changes go through Terraform + CI/CD.

## Application Deploy (DABs)

- Databricks resources (DLT pipelines, workflows, dashboards, alerts) are deployed via **Databricks Asset Bundles (DABs)** using `databricks bundle deploy`.
- Bundle definition: `dabs/databricks.yml` with target-specific overrides (`dev`, `hml`, `prod`).
- **DEV**: deploy locally via `make dabs_deploy_dev` or `databricks bundle deploy --target dev`.
- **HML + PROD**: deploy only via CI/CD (`deploy_databricks.yml`). Never deploy manually.
- DABs CLI is the Databricks-recommended approach for deploying pipelines and workflows. Terraform is used only for workspace infrastructure (clusters, external locations, metastore).

## Application Deploy (Streaming Apps)

- Streaming apps (`onchain-stream-txs`) are containerized and deployed to ECS Fargate.
- **DEV**: Docker Compose locally (`make deploy_dev_stream`).
- **HML**: ephemeral ECS tasks with RC image, tested for 10 min, then torn down.
- **PROD**: ECS services updated with promoted image via `deploy_streaming_apps.yml`.

## Application Deploy (Lambda Functions)

- Lambda functions are deployed via Terraform (`services/prd/10_lambda/`) with the `deploy_lambda_functions.yml` workflow.
- Handlers: `lambda/contracts_ingestion/` (EventBridge hourly → Etherscan → S3) and `lambda/gold_to_dynamodb/` (S3 event → DynamoDB sync).
- Lambda layer (`dm_chain_utils_layer.zip`) is pre-built in CI; handler zips are auto-generated by Terraform's `archive_file`.
- HML testing uses ephemeral Lambda functions created/destroyed per CI/CD run.

## Promotion Flow

```
feature/* → PR → develop → CI/CD trigger (manual dispatch) →
  HML deploy + integration test → release/* branch (auto) →
  PROD deploy (manual approval via GitHub Environment) → git tag
```

## S3 Prefix Convention

- All Firehose delivery uses `raw/` prefix: `s3://{bucket}/raw/{stream-name}/year=YYYY/month=MM/day=DD/hour=HH/`
- App logs Firehose: `s3://{bucket}/raw/app_logs/year=YYYY/month=MM/day=DD/hour=HH/`
- Gold exports: `s3://{bucket}/exports/`
- **Never** use `bronze/` as an S3 prefix. The `bronze/` naming is reserved for the Databricks DLT layer (Unity Catalog schema), not S3 paths.

## Secrets

- All secrets are stored in **GitHub Secrets** (repository or environment level).
- API keys for Etherscan are stored in **AWS SSM Parameter Store** (`/etherscan-api-keys`).
- Databricks DEV uses **PAT** tokens; PROD uses **OAuth M2M** (service principal).
- **Never** hardcode secrets in code, Terraform, or workflow files.
